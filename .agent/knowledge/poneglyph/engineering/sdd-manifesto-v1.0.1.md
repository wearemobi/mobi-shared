# M.O.B.I.™ · SDD Manifesto & Workflow v1.0.1

> Spec Driven Development: The Engineering Standard for Autonomous Execution

## 1. SDD Core Principles
The **Spec-First** philosophy is the absolute driver for engineering and startup management at **M.O.B.I.™**, replacing traditional project management overhead with precise documentation.

* **Single Source of Truth (SSoT):** Documentation is the highest authority. If a feature, fix, or rule is not in an approved SPEC, it does not exist.
* **Spec-First Mandate:** No code can be written without an approved SPEC. A GitHub Issue must exist *before* the corresponding SPEC is drafted.
* **Design for Sustainability (SSD):** Every specification must prioritize long-term maintenance, resource efficiency, and AI-readability.

## 2. Workflow Lifecycle
The development cycle follows a strict linear flow to guarantee autonomous, AI-oriented execution:

1.  **Issue Creation:** Define the exact **Context & Goal** and the **Definition of Done (DoD)** in GitHub.
2.  **SPEC-Drafting:** Generate a Markdown document outlining the technical or administrative solution.
3.  **Review & Approval:** The SPEC is reviewed and merged into the `main` branch (using **Rebase and merge**) *before* any implementation starts.
4.  **AI/Human Execution:** Developers and AI Agents ingest the approved SPEC to build the logic, architecture, or documentation.
5.  **Validation:** Code changes or assets are submitted via Pull Request (using **Squash and merge**) and verified strictly against the SPEC's **Acceptance Criteria**.

## 3. Guidelines for AI & Agentic Workflows
To ensure high-fidelity execution by LLMs and Autonomous Agents:

* **Context Binding:** Agents must be provided with the approved SPEC and the **Git Convention** as the primary constraints before generating any output.
* **Zero Hallucination Policy:** Agents are strictly forbidden from inventing features, assuming unwritten requirements, or applying stylistic changes outside the bounds of the provided SPEC.
* **Traceability:** Every AI-generated output must reference the SPEC version it originated from.

## 4. Standardized Artifacts
All project tasks must utilize the **SPEC or DOCS change request v1.0.1** template. This ensures a uniform structure:
* **Context & Goal:** The "Why" and the "What".
* **Implementation Plan:** The "How".
* **Acceptance Criteria:** The "Evidence of Success".
